# Hallucination Hunters — Freelancer Admin Chatbot

## System Documentation

---

## AI Agent Structure & Implementation Summary

This project implements a **multi-agent orchestrator architecture** where a central routing layer (`app.py`) coordinates three specialized AI agents, each responsible for a distinct domain of freelancer administration.

### Orchestrator Pattern

The system uses an **intent-based router** as the orchestrator. When a user message arrives, the orchestrator:

1. **Classifies intent** using an LLM-based classifier (`intent.py`) that categorizes messages into one of 5 intents (PROPOSAL, INVOICE, REMINDER, QUERY, GENERAL).
2. **Dispatches to the appropriate agent** and maintains a **stateful session** so that multi-turn conversations are routed to the same agent until the task completes.
3. **Resets state** once the agent reports completion, making the orchestrator available for the next request.

### Agent Responsibilities

| Agent | File | Role | LLM Usage |
|-------|------|------|-----------|
| **Intent Classifier** | `intent.py` | Classifies user intent into 5 categories | Zero-shot classification (temperature=0, max 10 tokens) |
| **Proposal Agent** | `proposal_agent.py` | Collects project details, generates professional proposals, exports PDF + DOCX | Data extraction + document generation (temperature=0.7, max 2000 tokens) |
| **Invoice Agent** | `invoice_agent.py` | Collects billing details, calculates totals, generates invoice PDFs | Data extraction (temperature=0, max 800 tokens) |
| **Reminder Agent** | `reminder_agent.py` | Detects overdue tone, drafts payment reminder emails, sends via Gmail | Email generation (temperature=0.6, max 800 tokens) |

### How Each Agent Works

All three task agents follow a common **Extract → Validate → Execute** pattern:

```
User Message
    │
    ▼
┌─────────────────────────┐
│  LLM Data Extraction    │  GPT-4o-mini parses natural language
│  (structured JSON out)  │  into structured field values
└───────────┬─────────────┘
            ▼
┌─────────────────────────┐
│  Field Validation       │  Check required fields; if missing,
│  (missing check)        │  ask user for remaining info
└───────────┬─────────────┘
            ▼
┌─────────────────────────┐
│  Task Execution         │  Generate document / calculate totals /
│  (LLM + business logic) │  draft email / send via SMTP
└───────────┬─────────────┘
            ▼
┌─────────────────────────┐
│  Persist & Respond      │  Save to SQLite, export files,
│                         │  return result to user
└─────────────────────────┘
```

### Key Design Decisions

- **Single LLM model** (`gpt-4o-mini`) used across all agents for cost efficiency
- **Session-based state** — each user session tracks the active intent, conversation history, and extracted fields in-memory
- **Template-based input** — agents present a full-field template for single-shot data entry (rather than multi-turn Q&A per field)
- **Tone-aware reminders** — automatically adapts email tone (Gentle / Firm / Urgent) based on overdue duration
- **Multi-language output** — LLM receives language instructions to generate responses and documents in the user's selected language
- **No hardcoded responses** — all substantive content (proposals, emails, general chat) is generated by the LLM at runtime

## 2. Intent Detection & Request Routing

### How Intent Detection Works

Intent detection uses an **LLM-based zero-shot classifier** implemented in `intent.py`. When a user sends a message to the chatbot and no active flow is in progress, the system classifies the user's intent.

#### Classification Process

1. The user's message is sent to GPT-4o-mini with a **system prompt** defining 5 possible labels:

| Intent | Description |
|--------|-------------|
| `PROPOSAL` | User wants to create a project proposal |
| `INVOICE` | User wants to generate an invoice |
| `REMINDER` | User wants to send a payment reminder email |
| `QUERY` | User asks about existing invoices, clients, or data |
| `GENERAL` | Greeting, chit-chat, or anything else |

2. The LLM is set to `temperature=0` with `max_tokens=10` to ensure deterministic, single-word output.
3. The response is validated against the known labels. Invalid responses default to `GENERAL`.

#### Number-Based Shortcuts

Before invoking the LLM, the router checks for **number-based shortcuts** (1–4). This allows quick menu-style selection:

| Number | Mapped Intent | Replacement Message |
|--------|---------------|---------------------|
| `1` | `PROPOSAL` | "I want to create a proposal" |
| `2` | `INVOICE` | "I want to create an invoice" |
| `3` | `REMINDER` | "I want to send a payment reminder" |
| `4` | `QUERY` | "Show me my invoices" |

#### Request Routing Flow

```
User Message
    │
    ├── Number shortcut (1-4)?  ──→ Map to intent directly
    │
    └── No shortcut ──→ LLM classification
                              │
                ┌─────────────┼─────────────────┐
                ▼             ▼                 ▼
           PROPOSAL       INVOICE           REMINDER
           QUERY          GENERAL
                │             │
                ▼             ▼
         Agent Handler    Direct Reply
         (stateful)       (single-turn)
```

#### Stateful Agent Dispatch

When an intent of PROPOSAL, INVOICE, or REMINDER is detected, it is stored in the session as `sess["intent"]`. All subsequent messages from that session are routed to the corresponding agent **until the flow completes** (all fields collected and document generated) — at which point `sess["intent"]` is reset to `None`.

## 3. Proposal Generation

### Prompt Structure & Flow

Proposal generation is handled by `proposal_agent.py` and follows a **multi-turn conversational flow**:

#### Step 1: Template Presentation

When a user initiates a proposal, the system presents a **single-shot template** with all fields:

```
Client Name:
Project Title:
Project Description:
Deliverables:
Timeline:
Your Name (Freelancer): Hallucination Hunters       ← pre-filled
Budget: (optional)
Your Skills: AI, Full Stack Development              ← pre-filled
```

#### Step 2: LLM-Based Data Extraction

Each user response is processed by the `_extract_fields()` function, which uses GPT-4o-mini with a data-extraction system prompt. The LLM parses natural language into structured JSON fields:

**Required Fields:**
- `client_name`, `project_title`, `project_description`, `deliverables`, `timeline`, `freelancer_name`

**Optional Fields:**
- `budget`, `freelancer_skills`

Extracted fields are **merged incrementally** — previously stored values are preserved and new values override them.

#### Step 3: LLM-Based Document Generation

Once all required fields are collected, the system generates a **professional proposal** via GPT-4o-mini using this structure prompt:

**Proposal Sections:**
1. Introduction
2. Project Understanding
3. Proposed Approach
4. Deliverables
5. Timeline
6. Pricing
7. Terms
8. Closing (includes company contact details)

If a non-English language is selected, the LLM is instructed to write the entire proposal in that language.

#### Step 4: Document Export

The generated markdown text is converted into two formats:
- **PDF** — via ReportLab (A4 page, with heading styles and paragraph formatting)
- **DOCX** — via python-docx (with proper heading levels and bullet lists)

Both files are saved to the `generated_docs/` directory and their paths are stored in the database.

## 4. Invoice Calculation & Numbering

### Invoice Flow

Invoice generation is handled by `invoice_agent.py` with a similar multi-turn conversational flow.

#### Input Template

```
Client Name:
Client Email:
Project Name:
Invoice Number: (leave blank for auto)
Invoice Date: 2026-03-07                             ← auto-filled with today
Due Date:
Work Items: (e.g. Web Development)
Hours per Item:
Rate: ($ per hour)
Tax / GST: (% or leave blank for none)
Freelancer Bank / Payment Details:
Notes: From Hallucination Hunters | hallucination_hunters@gmail.com | 08043672418
```

#### Invoice Number Auto-Generation

The `next_invoice_number()` function in `database.py`:

1. Fetches **all existing invoice numbers** from the database
2. Finds the **maximum numeric value** (starting baseline: 1000)
3. Returns `max + 1` as a string

Example: If invoices `1001`, `1002`, `1003` exist → next number is `1004`.

If the user provides a custom invoice number and it **doesn't already exist**, that number is used. Otherwise, auto-generation kicks in.

#### Calculation Logic (`_calc` function)

```python
For each work item:
    item.amount = hours × rate

subtotal = sum(all item amounts)
tax_amount = subtotal × (tax_rate / 100)
grand_total = subtotal + tax_amount
```

All amounts are rounded to 2 decimal places.

**Example:**

| Work Item | Hours | Rate | Amount |
|-----------|------:|-----:|-------:|
| Web Development | 20 | $30 | $600.00 |
| API Integration | 10 | $50 | $500.00 |

- **Subtotal:** $1,100.00
- **Tax (18%):** $198.00
- **Grand Total:** $1,298.00

#### PDF Generation

The invoice PDF is generated using ReportLab with:
- A4 page size with 2cm margins
- Professional table layout with dark header row (`#1e293b`)
- Grid lines, right-aligned numerical columns
- Payment instructions and notes sections

#### Overdue Detection

The `/api/invoices` endpoint performs **automatic overdue detection**:

```python
if invoice.status == "UNPAID" and invoice.due_date < today:
    invoice.status = "OVERDUE"     # Updated in response AND database
```

#### Mark as Paid

The `/api/invoice/<id>/mark-paid` POST endpoint allows manually updating an invoice status to `PAID`.

## 5. Reminder Tone Detection

### Tone-Based Email Generation

The reminder system automatically selects an appropriate email tone based on **how many days overdue** the invoice is. This is implemented in `reminder_agent.py`.

#### Tone Thresholds

| Overdue Duration | Tone | Description |
|------------------|------|-------------|
| **1–7 days** | **Gentle** | Friendly reminder, assumes the client simply forgot. Warm, non-pressuring tone. |
| **8–21 days** | **Firm** | Polite but direct. Makes clear the payment is overdue and expected promptly. |
| **22+ days** | **Urgent** | Professional but firm. States consequences such as work pause or late fees if applicable. |

#### Implementation

```python
def _detect_tone(days_overdue: int) -> str:
    if days_overdue <= 7:
        return "gentle"
    elif days_overdue <= 21:
        return "firm"
    return "urgent"  # 22+ days
```

#### Days Overdue Calculation

```python
due_date = datetime.strptime(invoice["due_date"], "%Y-%m-%d")
days_overdue = max((datetime.now() - due_date).days, 1)  # minimum 1 day
```

#### Reminder Generation Flow

```
Invoice Identified
       │
       ▼
Calculate days_overdue
       │
       ▼
_detect_tone(days_overdue)  →  "gentle" | "firm" | "urgent"
       │
       ▼
_generate_reminder(invoice_data, tone, days_overdue)
       │  (GPT-4o-mini generates email with tone-specific instructions)
       ▼
Returns { "subject": "...", "body": "..." }
       │
       ▼
User previews draft → confirms "send" → email sent via Gmail
```

#### Tone Description Passed to LLM

The detected tone's description is injected into the system prompt:

- **Gentle:** *"Friendly reminder, assumes the client simply forgot. Warm, non-pressuring tone."*
- **Firm:** *"Polite but direct. Makes clear the payment is overdue and expected promptly."*
- **Urgent:** *"Professional but firm. States consequences such as work pause or late fees if applicable."*

## 6. Gmail Integration & Email Sending

### Setup Requirements

Gmail integration requires two credentials stored in `API.txt`:

```
GMAIL_SENDER=your_email@gmail.com
GMAIL_APP_PASS=xxxx xxxx xxxx xxxx
```

> **Note:** `GMAIL_APP_PASS` must be a **Google App Password**, not the regular Gmail password. To generate one:
> 1. Go to Google Account → Security → 2-Step Verification (must be enabled)
> 2. Go to App Passwords → Generate a new app password for "Mail"
> 3. Copy the 16-character password into `API.txt`

### Email Sending Mechanism

The `send_email()` function in `reminder_agent.py` uses Python's `smtplib` over **SSL (port 465)**:

```python
def send_email(sender, app_pass, to_email, subject, body):
    msg = EmailMessage()
    msg["From"] = sender
    msg["To"] = to_email
    msg["Subject"] = subject
    msg.set_content(body)

    context = ssl.create_default_context()
    with smtplib.SMTP_SSL("smtp.gmail.com", 465, context=context) as smtp:
        smtp.login(sender, app_pass)
        smtp.send_message(msg)
```

### Two Entry Points for Sending Reminders

#### 1. Via Chat (Conversational Flow)

1. User types "3" or "send a payment reminder"
2. System asks for invoice number
3. User provides invoice number → system generates draft with auto-detected tone
4. User reviews the draft and types "send" to confirm
5. Email is sent and reminder is logged in the database

#### 2. Via Payment Status Tab (Direct Button)

1. User clicks "Send Reminder" button on an invoice card in the Payment Status tab
2. Frontend calls `POST /api/send-reminder/<invoice_number>`
3. Backend automatically:
   - Looks up invoice and client email
   - Calculates days overdue
   - Detects tone
   - Generates reminder via LLM
   - Sends email
   - Logs reminder in database
4. Success/failure response is shown to the user
